What It Is
A Python library and AWS-native platform for SR 11-7 compliant model risk management. Covers the full
validation lifecycle: initial validation, ongoing monitoring, fairness analysis, stress testing, and regulatory
documentation — built for financial services and insurance, where model governance is a regulatory
requirement.
Why this gap exists: Existing tools (Evidently, WhyLabs, SageMaker Model Monitor) solve data drift and
operational monitoring. None are designed around the regulatory validation workflow — the challenger
model comparison, conceptual soundness framework, tiered risk classification, and immutable audit trail.
This toolkit fills that gap.

Phase 1 — Core Library Foundation

3 Weeks

Goal: A pip-installable package that produces rigorous metrics with proper uncertainty quantification. No
AWS, no dashboard yet — something usable at work immediately.

Week 1 — Scaffolding + Classification Metrics
• pyproject.toml with proper packaging (hatchling)
• GitHub Actions: lint (ruff), type check (mypy), test (pytest), coverage gate
• classification.py : AUC-ROC with bootstrap CI, KS statistic + p-value, GINI coefficient, lift table,
cap curve, precision-recall AUC
• calibration.py : reliability diagram, ECE, Brier score decomposition (reliability + resolution +
uncertainty)
• Full unit test suite with synthetic data fixtures

Week 2 — Regression + Statistical Tests
• regression.py : MAE, RMSE, MAPE, Theil's U, residual diagnostics (heteroscedasticity,
autocorrelation)
• drift.py : PSI (the workhorse of insurance validation), CSI by feature, KS test for continuous
features, chi-square for categorical, Jensen-Shannon divergence
• corrections.py : Bonferroni and Benjamini-Hochberg FDR control — critical when testing dozens of
features simultaneously

Week 3 — Uncertainty Quantification + Challenger Comparison
• bootstrap.py : percentile bootstrap and BCa (bias-corrected accelerated) for any metric
• comparison.py : DeLong test for AUC comparison (exact, not bootstrap), two-sample tests on KS
and GINI, effect size alongside p-values

Phase 2 — Statistical Rigor + Fairness

3 Weeks

Goal: The module a regulator would actually care about.

Week 4 — Stability Analysis
• PSI by score decile (vintage analysis) — track population shifts over time
• PSI threshold alerts: green (<0.1), yellow (0.1–0.25), red (>0.25)
• Feature-level CSI for top N predictors
• Gini stability over time — plot Gini by monthly cohort
• Vintage analysis: performance curves by origination cohort

Week 5 — Fairness Module
• Disparate impact: 4/5ths rule (EEOC standard), demographic parity difference, equalized odds,
predictive parity, individual fairness metric
• Proxy detection: mutual information between features and protected attributes, permutation
importance with protected class correlation, redlining flag for geographic features correlated with
race/income
• Bootstrap CIs on all fairness metrics
• Intersectional fairness: performance by (race × gender) or (age × geography)

Week 6 — Sensitivity + Stress Testing
• Perturbation analysis: one-at-a-time feature shifts ±1 std, tornado diagram output
• SHAP stability: do feature importances shift across time splits?
• Stress scenarios: predefined macroeconomic shocks (2008 crisis, COVID, rate spike), custom
scenario API, expected loss under stress
Deliverable: Full SR 11-7-style statistical validation battery with properly corrected p-values and
bootstrap CIs throughout.

Phase 3 — Regulatory Compliance Layer

3 Weeks

Goal: Produce documentation that a model risk examiner would accept.

Week 7 — SR 11-7 Report Engine
Structured HTML/PDF report with all required sections:
1. Model overview and purpose
2. Conceptual soundness assessment
3. Data quality findings
4. Performance results (with CIs)
5. Benchmarking vs. challenger model
6. Limitations and assumptions
7. Ongoing monitoring plan
8. Conclusions and recommendations
Auto-populates quantitative sections from validation run. Narrative sections left for human input or LLM
fill-in (Phase 6).

Week 8 — Model Registry + Risk Tiering
• Model registry: PostgreSQL-backed inventory with model_id, owner, type, purpose, training_date,
validation_status, next_review_date, tier
• Risk tiering: Tier 1 (high — drives significant financial decisions), Tier 2 (moderate), Tier 3 (low /
immaterial) with auto-classification heuristics
• Audit trail: immutable append-only log of all validation actions with user, timestamp, and data hash

Week 9 — Model Card + Lineage
• Google Model Card format with financial services extensions, auto-populated from validation
metadata
• Feature provenance and dependency graph
• Executive summary — plain English version for non-technical stakeholders
Deliverable: Full regulatory documentation package auto-generated from a single validation run.

Competitive Advantage
Most people building this kind of tool implement generic ML metrics. Three specific advantages make
this different:
1. Regulatory fluency — knowing what examiners actually ask for, having answered those questions in
production at a Tier 1 insurer
2. Mathematical rigor — multiple testing correction, proper BCa bootstrap methods, and the DeLong
test for AUC comparison implemented correctly rather than naively
3. Insurance domain depth — PSI thresholds, vintage analysis, and the fairness issues specific to
insurance pricing understood at depth, not approximated
The combination of these three in one open-source package does not currently exist. That is the moat.
